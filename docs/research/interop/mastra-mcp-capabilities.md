## findings

Mastra has first-class support for exposing an app as an MCP SERVER via the `MCPServer` class in @mastra/mcp (server class lives in @mastra/mcp; its base class `MCPServerBase` and `MCPServerConfig` type live in @mastra/core/mcp). It exposes arbitrary `createTool` tools directly, auto-converts Agents to `ask_<key>` tools and Workflows to `run_<key>` tools, and supports stdio, legacy SSE, Hono SSE, and streamable HTTP (recommended) transports, including a stateless `serverless: true` mode. Per-request context reaches tools: each MCP call's transport `extra` (authInfo from `req.auth`, sessionId, requestInfo incl. headers) is copied into a fresh `RequestContext` and handed to the tool's execute as `context.requestContext` plus `context.mcp.extra` â€” that is the tenant seam. OAuth protection is provided by `createOAuthMiddleware` (RFC 9728). The newest release (1.14.0) adds `mapAuthInfoToUser` + per-server FGA mapping for tool-level authorization, but FGA is an EE feature requiring an FGA provider on the Mastra server; without it there is NO built-in per-tool authorization â€” every connected client can call every registered tool, so a governed workspace must enforce authorization inside each tool (secureToolWrapper-style) using authInfo from the context. Client side, `MCPClient` consumes external MCP servers (stdio subprocess or HTTP URL) and feeds them to agents via `listTools()` (static) or `listToolsets()` (per-request, explicitly documented for multi-tenant per-user credentials) â€” a Python sidecar works either as `command: 'python'` stdio or as a streamable-HTTP URL. Key repo facts: ActNG2 has @mastra/core 1.49.0 but @mastra/mcp is NOT installed there; YesChef has @mastra/mcp 1.8.0 installed but its production public MCP server (/Users/justinmorrey/YesChef/apps/yeschef-server/src/mcp/public/server.ts + http.ts) deliberately uses the raw @modelcontextprotocol/sdk McpServer with a hand-rolled stateless-per-request transport and Bearer PAT auth (rcp_mcp_* tokens, fail-clear 401), not Mastra's MCPServer.

2. Auth / per-request context: For every tools/call, the MCP SDK RequestHandlerExtra (`extra`) is copied key-by-key into a FRESH RequestContext (`createProxiedRequestContext`, dist/index.js ~line 4476) and passed to the tool as `context.requestContext`; the same extra is also at `context.mcp.extra`. extra contains: `authInfo` (whatever the host set on `req.auth` before delegating to startHTTP â€” SDK streamableHttp.js reads `req.auth`), `sessionId`, `requestInfo` (incl. HTTP headers), `signal`. Documented universal access pattern: `const mcpExtra = context?.mcp?.extra ?? context?.requestContext?.get('mcp.extra'); const authInfo = mcpExtra?.authInfo`. So the multi-tenant seam is: your HTTP middleware validates the bearer token, sets `req.auth = { tenantId, userId, ... }`, and every tool reads it from authInfo â€” the server does NOT inject it into tool args, tools must consume it themselves. `createOAuthMiddleware({ oauth: { resource, authorizationServers, validateToken, scopesSupported? }, mcpPath? })` implements RFC 9728: serves /.well-known/oauth-protected-resource, 401s with WWW-Authenticate, runs your validateToken(token, resource); helpers `createStaticTokenValidator(tokens)` and `createIntrospectionValidator(endpoint, {clientId, clientSecret})` (RFC 7662). IMPORTANT: the middleware returns `{ proceed, handled, tokenValidation }` â€” it does NOT set req.auth for you; you wire tokenValidation â†’ req.auth manually. Per-tool authorization exists only via FGA (EE): when the Mastra server has an `fga` provider, tools/list and tools/call check `requestContext.get('user')` (dynamic import of '@mastra/core/auth/ee'); 1.14.0's mapAuthInfoToUser bridges authInfo â†’ that user. No other documented multi-tenant MCP-server pattern; docs' multi-tenant guidance is on the CLIENT side (listToolsets per-request credentials).

## caveats

1) No built-in per-tool authorization outside EE FGA: absent an FGA provider, any authenticated (or, if you skip middleware, ANY) MCP client can list and call every registered tool. A governed reserving workspace must do authorization inside each tool from context.mcp.extra.authInfo â€” MCPServer never injects tenant/user into tool args (mirror of the secureToolWrapper rule). 2) Version gap: `mapAuthInfoToUser` and per-server `fga` config are in the current docs/1.14.0 but NOT in the 1.8.0 installed in YesChef â€” building the governed-auth story requires the newer package; docs describe latest, installed code lags. 3) createOAuthMiddleware validates tokens but leaves attaching identity to you: nothing sets `req.auth` automatically, and forgetting that wire-up means tools silently see authInfo === undefined (fail-open risk if tools don't hard-require it). 4) The proxied RequestContext is constructed FRESH from the transport `extra` per call â€” a RequestContext populated by your own upstream Mastra middleware does not flow through automatically; only extra's keys (authInfo, sessionId, requestInfo, signal, etc.) exist in it. 5) Agents-as-tools are free-text in (`{message}`) â†’ agent.generate() out: nondeterministic, no structured output contract, no documented per-tenant memory/thread scoping â€” poor fit for governed calculations; prefer exposing tools/workflows. Workflows-as-tools are fire-and-return (createRun().start()); suspend/resume/human-in-the-loop is not surfaced over MCP. 6) Stateful streamable-HTTP sessions are held in in-process maps â€” multi-instance deployment needs sticky sessions or `serverless: true` stateless mode, and stateless mode sacrifices elicitation, notifications, and resumability. 7) SSE transport is legacy; new work should use streamable HTTP only. 8) Node >=22.13.0 engine requirement on current @mastra/mcp. 9) Known TS papercut in this codebase: Zod + MCP SDK schema types trigger TS2589 deep-instantiation errors (the existing server.ts is @ts-nocheck for this reason) â€” expect the same with complex Zod schemas on exposed tools.
