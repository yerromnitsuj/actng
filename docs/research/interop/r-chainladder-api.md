# r-chainladder-api — source-grounded research (2026-07-17)

## findings

Researched the CURRENT ChainLadder R package (CRAN v0.2.21, published 2026-02-18, maintainer Markus Gesmann) from the CRAN package page and the live master-branch .Rd sources at github.com/mages/ChainLadder. All facts below are quoted from those sources, not memory.

Key design-relevant conclusions for an interchange adapter:
1. The triangle is a plain S3-classed numeric matrix (class "triangle") whose interchange-stable identity lives entirely in dimnames: rownames = origin periods, colnames = dev periods, and names(dimnames(Triangle)) conventionally c("origin","dev"). Long-format round-trip is first-class: as.triangle(data.frame, origin=, dev=, value=) builds it; as.data.frame.triangle / as.LongTriangle flatten it back. This long format (origin, dev, value) is the natural JSON interchange shape.
2. There is NO direct API to hand MackChainLadder a vector of selected development factors (no f= argument). Injection routes are: (a) the weights matrix (0/NA excludes individual link ratios), (b) CLFMdelta(Triangle, selected, tolerance=5e-4) which solves for per-period delta values such that coef(chainladder(Triangle, delta=delta0)) reproduces the selected factors (with a foundSolution attribute — not all selections are feasible), and (c) numeric overrides for tail (tail=<number>, tail.se=, tail.sigma=) and est.sigma=<numeric>. An adapter exporting externally-selected factors into ChainLadder must go through CLFMdelta+chainladder or accept that Mack std errors are only defined for factors the model itself can produce.
3. Nothing JSON-native exists in the package — jsonlite is not in Imports or Suggests. Serialization is entirely the adapter's job.
4. The package is actively maintained (release 5 months ago) with no explicit R-version floor declared in the master DESCRIPTION.

## apiFacts

== 1. Triangle data structure ==
- triangle(): `triangle(..., bycol=FALSE, origin="origin", dev="dev", value="value")` — builds the matrix from vectors, filling by row by default; vector argument names become dimnames, and names of elements of the first vector supply the other dimension's names.
- as.triangle(): generic with matrix and data.frame methods, shared args `origin="origin", dev="dev", value="value"`. The data.frame method converts long-format ("database") storage to a triangle; origin and dev columns must be numeric or numeric-convertible character; multiple records with the same (origin, dev) are AGGREGATED (summed) — an adapter must pre-aggregate or accept this.
- as.data.frame.triangle(): `as.data.frame(x, row.names=NULL, optional, lob=NULL, na.rm=FALSE, ...)` — long format out; na.rm drops the empty future cells; lob adds a line-of-business label column.
- as.LongTriangle(): `as.LongTriangle(Triangle, varnames=names(dimnames(Triangle)), value.name="value", na.rm=TRUE)` — like as.data.frame but stores origin/dev as FACTORS preserving dimension order (falls back to c("origin","dev") when dimnames are unnamed). Best flattening for label-order-preserving export.
- cum2incr(Triangle) and incr2cum(Triangle, na.rm=FALSE) convert between cumulative and incremental; both return triangles.
- ata(): `ata(Triangle, NArow.rm=TRUE, colname.sep="-", colname.order=c("ascending","descending"))` — matrix of individual link ratios with attributes `smpl` (simple-average factors) and `vwtd` (volume-weighted factors). Useful for exporting the raw factor field.

== 2. Key model functions ==
- MackChainLadder(Triangle, weights=1, alpha=1, est.sigma="log-linear", tail=FALSE, tail.se=NULL, tail.sigma=NULL, mse.method="Mack"). Returned object components: call, Triangle, FullTriangle, Models (per-dev-period lm objects), f (age-to-age factors incl. tail), f.se, F.se, sigma, Mack.ProcessRisk, Mack.ParameterRisk, Mack.S.E (per-origin total SE), Total.Mack.S.E (scalar across origins), Total.ProcessRisk, Total.ParameterRisk, weights, alpha, tail (includes the fitted lm when tail=TRUE). mse.method: "Mack" (default) or "Independence" (adds Murphy/BBMW cross-product terms).
- chainladder(Triangle, weights=1, delta=1): the underlying regression engine; returns list(Models, Triangle, weights, delta); use predict() on it for the completed triangle. Relationship: MackChainLadder alpha = 2 - delta (Mack 1999 notation).
- CLFMdelta(Triangle, selected, tolerance=.0005, ...): given a vector of user-selected age-to-age factors (one per dev period), returns per-period delta values such that coef(chainladder(Triangle, delta=delta0)) == selected within tolerance; result carries a logical attribute `foundSolution` per element. This is THE sanctioned path for external factor injection; infeasible selections exist.
- BootChainLadder(Triangle, R=999, process.distr=c("gamma","od.pois"), seed=NULL). Returns: call, Triangle, f, simClaims (m×n×R array), IBNR.ByOrigin (m×1×R), IBNR.Triangles (m×n×R), IBNR.Totals (length-R vector), ChainLadder.Residuals (adjusted Pearson), process.distr, R.
- MunichChainLadder(Paid, Incurred, est.sigmaP="log-linear", est.sigmaI="log-linear", tailP=FALSE, tailI=FALSE, weights=1). Returns: call, Paid, Incurred, MCLPaid, MCLIncurred (full forecast triangles), MackPaid, MackIncurred (embedded MackChainLadder fits), PaidResiduals, IncurredResiduals, QResiduals, QinverseResiduals, lambdaP, lambdaI, qinverse.f, q.f, rhoP.sigma, rhoI.sigma.
- ClarkLDF(Triangle, cumulative=TRUE, maxage=Inf, adol=TRUE, adol.age=NULL, origin.width=NULL, G="loglogistic"): needs >=4 columns; G in {"loglogistic","weibull"}; returns class "ClarkLDF" with method/growthFunction/PAR/THETAU/THETAG/SIGMA2/MAXAGE, per-origin Origin/CurrentValue/CurrentAge/FutureValue/ProcessSE/ParameterSE/StdError/Ldf/TruncatedLdf/GrowthFunction, Total list, per-observation origin/age/fitted/residuals/stdresid/FutureValueGradient, plus FI (Fisher info), loglik value, convergence counts.
- ClarkCapeCod(Triangle, Premium, cumulative=TRUE, maxage=Inf, adol=TRUE, adol.age=NULL, origin.width=NULL, G="loglogistic"): Premium is scalar or per-origin vector (recycled with warning); adds ELR to the ClarkLDF-style output.
- glmReserve(triangle, var.power=1, link.power=0, cum=TRUE, mse.method=c("formula","bootstrap"), nsim=1000, nb=FALSE, ...): var.power=1 = over-dispersed Poisson, NULL = estimate from data; link.power=0 = log link; nb=TRUE switches to negative binomial. Returns class "glmReserve": call, summary (data.frame of reserve stats), Triangle, FullTriangle, model (glm/cpglm fit), and when bootstrapping: sims.par, sims.reserve.mean, sims.reserve.pred.
- tweedieReserve(triangle, var.power=1, link.power=0, design.type=c(1,1,0), rereserving=FALSE, cum=TRUE, exposure=FALSE, bootstrap=1, boot.adj=0, nsim=1000, proc.err=TRUE, p.optim=FALSE, p.check=c(0,seq(1.1,2.1,by=0.1),3), progressBar=TRUE, ...). Returns: call, summary (Latest/Det.Reserve/Ultimate/Dev.To.Date plus extra fields when bootstrap>0 or rereserving), Triangle, FullTriangle, model, scale (phi), bias, GLMReserve, gamma_y (calendar-year factors when modeled), res.diag (residual diagnostics data.frame), distr.res_ult ("Ultimate View" full distribution when bootstrap>0), distr.res_1yr ("1yr View" when rereserving=TRUE).
- Tail estimation in Mack/Munich: tail=FALSE none; tail=TRUE estimates by log-linear extrapolation of the development factors; tail=<numeric> uses that factor directly; tail.se / tail.sigma NULL = log-linear-regression estimates, numeric = user-supplied overrides (only relevant when tail > 1).

== 3. Serialization ==
- Nothing JSON-native in the package: Imports = Matrix, actuar, methods, stats, lattice, grid, tweedie, utils, systemfit, statmod, cplm(>=0.7-3), ggplot2, MASS; Suggests = RUnit, knitr, rmarkdown. No jsonlite anywhere.
- Recommended interchange idiom: flatten via as.LongTriangle() (or as.data.frame(tri, na.rm=TRUE)) to a 3-column (origin, dev, value) data.frame and serialize that with jsonlite::toJSON(df, dataframe="rows") — this preserves origin/dev labels. Serializing the matrix directly with jsonlite loses dimnames (default matrix encoding is nested row-major arrays without names), so an adapter should carry origin/dev labels explicitly and reconstruct with as.triangle(df, origin=, dev=, value=). Fitted objects (MackChainLadder etc.) are plain named lists containing matrices/vectors plus non-serializable lm objects (Models, tail lm) — an adapter should whitelist the numeric components (f, f.se, F.se, sigma, Mack.S.E, Total.Mack.S.E, FullTriangle, process/parameter risk) and drop the model objects.

## conventionNotes

- alpha (MackChainLadder, per Mack 1999): alpha=1 (default) = historical chain-ladder / volume-weighted factors; alpha=0 = straight (simple) average of individual development factors; alpha=2 = ordinary regression of C_{k+1} on C_k. NOTE the trap: chainladder()'s delta uses the OPPOSITE-feeling scale with alpha = 2 - delta (delta=1 volume-weighted, delta=2 simple average, delta=0 regression through origin). An adapter must not conflate the two parameterizations.
- est.sigma: "log-linear" (default) estimates sigma_{n-1} by log-linear regression on the sigma series; "Mack" uses Mack's (1999) approximation; a numeric value injects a user sigma. Auto-fallback behavior: if the log-linear regression looks inappropriate (p-value > 0.05) it switches to "Mack" — so round-tripped results can silently differ from the requested method; adapters should record the effective method.
- weights matrix: same dimensions as Triangle, entries in [0,1]; default scalar 1 = all points included; 0 or NA drops that individual link ratio from factor estimation. This is the point-exclusion mechanism (e.g. excluding high/low or old diagonals), not a factor-override mechanism.
- Tail: tail=TRUE = log-linear extrapolation estimate, tail=<number> = exact user factor; tail.se/tail.sigma numeric overrides apply only when the tail factor > 1. Munich splits these per triangle as tailP/tailI (and est.sigmaP/est.sigmaI), forwarded to the embedded MackChainLadder calls.
- as.triangle data.frame method aggregates duplicate (origin, dev) rows and requires numeric-convertible origin/dev columns; as.LongTriangle emits factors (order-preserving) while as.data.frame.triangle emits plain columns with merged rownames by default.
- MackChainLadder expects a CUMULATIVE m×n triangle with m >= n; Clark functions accept cumulative or incremental via cumulative=; glmReserve/tweedieReserve accept either via cum=.

## caveats

- Maintenance: actively maintained — CRAN v0.2.21 released 2026-02-18 (5 months ago), maintainer Markus Gesmann, repo mages/ChainLadder, docs at https://mages.github.io/ChainLadder/. In the CRAN ActuarialScience and Finance task views. License GPL(>=2).
- R version floor: the master-branch DESCRIPTION declares NO explicit `Depends: R (>= ...)` line, so the effective floor is inherited from its dependency chain (ggplot2, cplm >= 0.7-3, tweedie, systemfit, etc.), which in practice means a reasonably modern R; an adapter should not assume an old-R guarantee and should pin against 0.2.x behavior, not a specific R release.
- The CRAN PDF reference manual is served compressed and did not extract cleanly via automated fetch; all signatures above were taken from the live man/*.Rd sources on GitHub master (which corresponds to the 0.2.21 release cycle). If pinning exactly to the CRAN release, verify against the 0.2.21 tag rather than master before freezing the adapter schema.
- tweedieReserve's Value section describes the returned object as class "glm" (unlike glmReserve's dedicated "glmReserve" class) — treat its extra components (summary, FullTriangle, distr.res_ult, ...) as list members bolted onto a glm fit when adapting.
- Fitted objects embed lm/glm model objects (Models, tail lm, model) that are not portably serializable; interchange should carry only the numeric summaries.
- WebFetch summarization is a small-model paraphrase of the .Rd files; exact default-argument spellings were quoted where possible, but any load-bearing signature should be re-verified with args() in an R session before code generation.
