# chainladder-python-api — source-grounded research (2026-07-17)

## findings

Researched chainladder-python v0.9.2 (latest on PyPI; casact/chainladder-python master) directly from GitHub source (raw files: chainladder/core/triangle.py, core/io.py, core/pandas.py, development/development.py, development/base.py, development/constant.py, development/clark.py, development/outstanding.py, development/munich.py, methods/mack.py, methods/bornferg.py, methods/benktander.py, methods/capecod.py, tails/curve.py, adjustments/bootstrap.py, chainladder/__init__.py, pyproject.toml) plus the GitHub releases and PyPI pages. RTD "latest" pages 404'd via fetch, so all API facts below are sourced from actual master-branch source code, which is the ground truth. Key takeaways for a bidirectional interchange adapter: (a) Triangle HAS a first-class to_json/read_json round-trip with a documented-in-code schema (metadata + pandas orient="split" data + recursive sub_tris/dfs) — this is the natural interchange target; (b) fitted-estimator JSON serialization captures ONLY hyperparameters (get_params), NOT fitted arrays (ldf_/sigma_), so exporting selections from chainladder means reading fitted Triangle attributes (ldf_, cdf_, sigma_, std_err_) and converting via to_frame(); (c) injecting externally-chosen factors INTO chainladder is first-class via DevelopmentConstant(patterns={age_months: factor}, style='ldf'|'cdf') — a dict keyed by development age in months; (d) several numeric conventions (log-linear sigma interpolation default, ODP bootstrap per Shapland with hat-matrix adjustment, drop_high rank semantics) will NOT match a vanilla Mack-1993/England-2002 implementation unless deliberately aligned. Sources: [Triangle source](https://github.com/casact/chainladder-python/blob/master/chainladder/core/triangle.py), [io.py](https://github.com/casact/chainladder-python/blob/master/chainladder/core/io.py), [core README](https://github.com/casact/chainladder-python/blob/master/chainladder/core/README.md), [releases](https://github.com/casact/chainladder-python/releases), [polars interchange issue #474](https://github.com/casact/chainladder-python/issues/474), [PyPI](https://pypi.org/project/chainladder/), [RTD Triangle docs](https://chainladder-python.readthedocs.io/en/latest/user_guide/triangle.html).

## apiFacts

== 1. Triangle class (chainladder/core/triangle.py) ==
Constructor: Triangle(data=None, origin=None, development=None, columns=None, index=None, origin_format=None, development_format=None, cumulative=None, array_backend=None, pattern=False, trailing=True, *args, **kwargs). `data` is a long-format pandas DataFrame (also accepts DataFrame-interchange-protocol objects or dict); `origin`/`development`/`columns`/`index` name columns in that frame. `origin_format`/`development_format` are strptime strings for date parsing. `cumulative: Optional[bool]` — None is allowed at construction but deprecation-warned ("in a future release this will result in an error"); required for grain()/dev_to_val().
Internal representation: a 4-D values array with axes (kdims, vdims, odims, ddims) = (index, columns, origin, development); shape = (len(kdims), len(vdims), len(odims), len(ddims)). odims = numpy datetime64 array of origin-period starts; ddims = integer month-lags for a development triangle OR DatetimeIndex for a valuation triangle. Backends: 'numpy', 'sparse' (pydata/sparse COO), 'cupy'; auto-sparsification controlled by options.AUTO_SPARSE; switch with set_backend().
Grain: origin_grain and development_grain are single chars 'M','Q','S','Y', INFERRED from the unique origin/development dates (not passed explicitly); `trailing=True` infers period-end month from the data (fiscal/trailing calendars supported), False assumes December year-end. Development grain is coarsened to max(origin_grain, development_grain). grain(grain='', trailing=False, inplace=False) re-grains with the 'OYDY'/'OQDM'-style string (origin grain then development grain); requires cumulative.
valuation_date: pd.Timestamp = max development/valuation date in the data. Sentinel options.ULT_VAL (2261-12-31) marks ultimate columns.
Cumulative/incremental: is_cumulative property; incr_to_cum(inplace=False) (cumsum along dev axis; cumprod for pattern triangles), cum_to_incr(inplace=False) (diff; division for pattern triangles). dev_to_val()/val_to_dev() convert between development-lag and valuation-period layouts and are exact inverses. latest_diagonal property = cells at max valuation per origin.
Missing cells: future/invalid cells are NaN; nan_triangle property is the valid-cell mask; conversions re-multiply by nan_triangle to restore NaN boundaries. Missing observed cells are also NaN (to_json fills with 0 on export — see below).
Multi-segment: yes — the index axis (kdims) is n-level (e.g., LOB x company), and vdims holds multiple measure columns (paid/incurred/counts); slicing is pandas-like (tri['paid'], tri.loc[...], groupby on index levels).

== 2. Serialization (chainladder/core/io.py + core/pandas.py) ==
YES — Triangle has to_json() (TriangleIO mixin) and module-level cl.read_json(); also cl.read_pickle() with to_pickle(path) implemented via dill.dump. to_json emits a JSON string with keys: {"metadata": {is_val_tri, is_cumulative, is_pattern, columns(serialized)}, "data": <triangle converted via cum_to_incr().dev_to_val().to_frame(keepdims=True, origin_as_datetime=True).fillna(0) then pandas to_json(orient="split", date_unit="ns")>, "sub_tris": {recursively serialized child TriangleIO objects}, "dfs": {pandas DataFrames/Series as JSON, Series via to_frame()}}. So the wire format is: incremental values, valuation (not lag) layout, NaN→0, pandas split-orient — this IS the closest thing to a documented interchange format and read_json round-trips it. Estimators: EstimatorIO.to_json emits {"params": get_params(deep=False), "__class__": class name}, recursing into nested BaseEstimators — i.e. hyperparameters only, NO fitted state; fitted state persists only via dill pickle. to_frame(origin_as_datetime=True, keepdims=False, implicit_axis=False) → DataFrame/Series; keepdims keeps all 4 dims as columns; implicit_axis adds the implicit valuation axis. Pandas passthrough methods on Triangle include to_clipboard, to_csv, to_excel, to_html, to_dict, unstack, pivot, melt, describe. No parquet method on Triangle itself (go through to_frame()). 
== 3. Estimators (sklearn BaseEstimator style: fit(X, y=None, sample_weight=None), fit_transform, get_params/set_params; fitted attrs end in underscore) ==
Development(n_periods=-1, average='volume', sigma_interpolation='log-linear', drop=None, drop_high=None, drop_low=None, preserve=1, drop_valuation=None, drop_above=np.inf, drop_below=0.0, fillna=None, groupby=None). average ∈ {'volume','simple','regression','geometric'} or numeric (Zehnwirth-Barnett: numeric delta with weight exponent (2-average)); source mapping in development/base.py: {"regression": 0, "volume": 1, "simple": 2} then w = sample_weight / X**exponent in WeightedRegression(axis=2, thru_orig=True) — so volume ⇒ Σ(next)/Σ(prior) classic chainladder, simple ⇒ arithmetic mean of link ratios, regression ⇒ unweighted OLS through origin (Σxy/Σx²). n_periods = number of most-recent origin periods in the average (-1 = all). drop = (origin, development) tuple or list of tuples (drops specific link ratios); drop_high/drop_low = int (N highest/lowest RANKED link ratios per development column across origins; bool ⇒ 1) with `preserve` = minimum surviving link-ratio count (violations relax the drop with a warning); drop_valuation = date-convertible str/list dropping whole diagonals; drop_above/drop_below = numeric bounds on link ratios. Fitted: ldf_, cdf_, sigma_, std_err_, std_residuals_ (Barnett-Zehnwirth weighted standardized residuals), w_, plus internals. All fitted attrs are themselves Triangle objects (extract via .to_frame() / .values) — fully programmatically extractable.
TailCurve(curve='exponential', fit_period=(None,None), extrap_periods=100, errors='ignore', attachment_age=None, reg_threshold=(1.00001, None), projection_period=12). curve='exponential' regresses ln(ldf−1) on age; 'inverse_power' regresses ln(ldf−1) on ln(age). fit_period = (start, stop) ages or boolean list selecting which LDFs enter the regression. Fitted: tail_, ldf_, cdf_, slope_, intercept_.
MackChainladder() — NO constructor parameters; all selection lives in the upstream Development transform (it consumes X's ldf_/sigma_/std_err_). Fitted: full_std_err_ (per-cell sigma/sqrt(c^(2-alpha))), process_risk_, parameter_risk_, total_process_risk_ (sqrt of quadrature sum across origins), total_parameter_risk_, mack_std_err_ = sqrt(process²+parameter²) per origin, total_mack_std_err_, summary_ (Latest/IBNR/Ultimate/Mack Std Err), plus ultimate_, ibnr_, full_triangle_ from Chainladder. Variance recursion in code: t_tot = sqrt(t1² + (ldf·risk)²) per Mack 1993.
BootstrapODPSample(n_sims=1000, n_periods=-1, hat_adj=True, drop=None, drop_high=None, drop_low=None, drop_valuation=None, random_state=None) — lives in chainladder.adjustments (adjustments/bootstrap.py), exported at top level. Produces resampled_triangles_ (n_sims stacked on the index axis) + scale_ (dispersion φ), design_matrix_, hat_. Usage: sims = cl.BootstrapODPSample(n_sims=N).fit_transform(tri); cl.Chainladder().fit(sims).ibnr_ gives the distribution. Process variance added by gamma sampling: random_state.gamma(shape=|m|/scale_, scale=scale_).
BornhuetterFerguson(apriori=1.0, apriori_sigma=0.0, random_state=None): apriori is a SCALAR MULTIPLIER on sample_weight; the exposure/apriori-ultimate vector itself is passed as sample_weight to fit(X, y=None, sample_weight=exposure_triangle). Fitted: ultimate_, ibnr_.
Benktander(apriori=1.0, n_iters=1, apriori_sigma=0, random_state=None): n_iters=1 ≡ BF exactly; n_iters→∞ converges to Chainladder; fitted ultimate_, ibnr_, expectation_ (computed aprioris).
CapeCod(trend=0, decay=1, n_iters=1, apriori_sigma=0.0, random_state=None, groupby=None): apriori is DERIVED from data (Σ weighted trended latest / Σ used-up exposure) with decay matrix decay^|origin_i−origin_j|; decay=1 ⇒ traditional Cape Cod, decay→0 ⇒ Chainladder. Fitted: apriori_, detrended_apriori_, ultimate_, ibnr_. A Trend transformer on X overrides the trend arg.
ClarkLDF(growth='loglogistic'|'weibull', groupby=None): MLE curve fit per Clark 2003 on incrementals; fitted theta_, omega_, elr_ (only when sample_weight=premium supplied ⇒ Cape Cod mode), ldf_, cdf_, norm_resid_.
CaseOutstanding(paid_to_incurred=('paid','incurred') tuple, paid_n_periods=-1, case_n_periods=-1, groupby=None): needs a Triangle with both paid and incurred vdims; fitted paid_ldf_, case_ldf_, case_to_prior_case_, paid_to_prior_case_, ldf_ (implied multiplicative patterns, per-origin).
MunichAdjustment(paid_to_incurred=None (required tuple/list of tuples), fillna=False): fitted lambda_, ldf_ (bivariate patterns), basic_cdf_, cdf_, rho_sigma_.
Pipelines: cl.Pipeline (sklearn-style steps), cl.GridSearch, cl.VotingChainladder all exist (chainladder/workflow/). Other exports: Chainladder, ExpectedLoss, TweedieGLM, BarnettZehnwirth, TailConstant, TailBondy, TailClark, BerquistSherman, ParallelogramOLF, Trend, TrendConstant, DisposalRate, read_csv/read_pickle/read_json/concat/load_sample/list_samples/minimum/maximum, options.

== 4. Injecting external factors ==
First-class: DevelopmentConstant(patterns=None, style='ldf', callable_axis=0, groupby=None) — "An Estimator that allows for including of external patterns into a Development style model." patterns = dict {age_in_months: factor}, e.g. {12: 4.0, 24: 2.9, ...}, or a callable returning such a dict per element along callable_axis (index or columns axis, enabling per-segment patterns). style='ldf' (age-to-age) or 'cdf' (age-to-ultimate; internally decomposed). Usage: cl.DevelopmentConstant(patterns=p, style='cdf').fit_transform(tri) then feed to any IBNR method. This is THE mechanism for pushing externally-selected factors into chainladder pipelines (it produces ldf_/cdf_ like any Development transform; note it carries no sigma_/std_err_, so MackChainladder can't run on top of it). Alternative: Triangle(pattern=True) can represent a pattern triangle directly, and drop/drop_valuation/n_periods on Development express selection-by-exclusion.

## conventionNotes

- Averaging: 'volume' = weighted regression through origin with w=1/x (⇒ Σnext/Σprior, i.e. dollar-weighted), 'simple' = w=1/x² (⇒ straight arithmetic mean of individual link ratios), 'regression' = w=1 OLS through origin (Σxy/Σx², the Zehnwirth-Barnett delta=0 case). Numeric average values use weight exponent (2-average). 'geometric' appears in the typed signature but is not in the {"regression":0,"volume":1,"simple":2} exponent map — it is handled separately; verify against installed source before relying on it.
- n_periods counts ORIGIN periods (rows), taking the most recent n link ratios per development column (window ends at each column's latest available ratio); -1 = all. This matches the common "n-year volume-weighted" convention but interacts with drop_*: rank-based drop_high/drop_low are applied per development column ACROSS origins after n_periods windowing, and `preserve` silently relaxes drops (with a warning) when fewer than `preserve` ratios would survive — a straight reimplementation that hard-drops will diverge.
- Mack: variance recursion follows Mack 1993 (t_tot = sqrt(prior² + (ldf·risk)²); mack_std_err_ = sqrt(process²+parameter²); total_mack_std_err_ > quadrature sum due to correlated parameter risk). BUT Development's default sigma_interpolation='log-linear' — Mack's 1993 paper's ad-hoc last-column rule sigma²_{n} = min(sigma⁴_{n-1}/sigma²_{n-2}, min(sigma²_{n-1}, sigma²_{n-2})) is the NON-default 'mack' option. A vanilla Mack-1993 implementation will not match chainladder's defaults on the last development column's sigma (and hence tail-adjacent std errs) unless sigma_interpolation='mack' is set. full_std_err_ uses sigma/sqrt(c^(2-alpha)) where alpha tracks the averaging choice.
- ODP bootstrap: implements the Shapland practitioner's-guide variant ("Using the ODP Bootstrap Model: A Practitioners Guide"), built on England-Verrall foundations: unscaled Pearson residuals (a−m)/sqrt|m|, standardized by HAT-MATRIX adjustment by default (hat_adj=True) rather than the simpler England-2002 degrees-of-freedom factor sqrt(n/(n−p)) (that's the hat_adj=False path); scale φ = Pearson χ²/dof; process variance via gamma(shape=|m|/φ, scale=φ) sampling (gamma approximation to ODP, keeps sign for negative incrementals). A straight England-2002 implementation differs in (a) residual standardization (dof vs hat), (b) process-error distribution (England's later papers use gamma too, but England-Verrall 2002's basic version is often implemented with ODP/Poisson draws), so numbers match only in distribution, never sim-for-sim, and means differ slightly under hat_adj.
- to_json wire format normalizes to INCREMENTAL + VALUATION layout with NaN→0 — an adapter must treat 0 vs missing carefully on re-import (read_json reverses it using the metadata flags is_val_tri/is_cumulative/is_pattern).
- Triangles are period-start indexed (odims are datetime64 period starts); development ddims are age-in-months integers (month-lags), even for annual triangles (12, 24, 36...). DevelopmentConstant pattern dicts are keyed by age in MONTHS.
- BF apriori convention: apriori is a multiplier, the actual exposure/apriori-ultimate vector travels in fit's sample_weight (itself a Triangle). CapeCod computes its own apriori from data; trend/decay are the knobs.

## caveats

- Version: latest released = 0.9.2 on PyPI (requires-python >=3.10); master pyproject also says 0.9.2 but has moved (pandas >=2.3.3,<=3.0.3; numpy >=2.0; scikit-learn >1.4.2; sparse >=0.9; patsy >=1.0.2; matplotlib; dill). The two page-summaries disagreed on the 0.9.2 release date (May 2024 vs May 2026) — treat the date as unverified, the version pin 0.9.2 as solid.
- Heavy pandas coupling: constructor consumes pandas DataFrames (plus DataFrame-interchange-protocol objects — polars via interchange was requested in issue #474 but pandas remains the API surface); to_json rides pandas to_json(orient='split', date_unit='ns'). An interchange adapter should target the to_json schema and/or to_frame(keepdims=True) long format, not internals.
- Fitted estimators do NOT serialize fitted state to JSON (params only); full fidelity persistence is dill pickle — not a cross-language interchange format. Bidirectional adapters must re-fit or export fitted Triangle attributes explicitly.
- cumulative=None at construction is deprecation-warned and slated to become an error — always set it explicitly.
- In flux: no concrete public v1.0 roadmap found (searched); active areas are dataframe-interchange/polars support and type-hint modernization (the typed signatures with Literal types are recent). The io schema has no formal version field — pin the library version in any adapter contract.
- RTD 'latest' doc pages returned 404 to programmatic fetch during this research; facts above came from master-branch source, which may be slightly ahead of the 0.9.2 wheel. Verify against the installed package's source for exact behavior of 'geometric' average and drop-rank tie-breaking before locking adapter semantics.
- All page reads were mediated by a summarizer model over fetched source; signatures quoted match the source structure but exact default spellings (e.g. reg_threshold=(1.00001, None)) should be spot-checked against the installed .py files if load-bearing.
